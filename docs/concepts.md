# How the repo works

## Source, assets, and evidence

The repo keeps three kinds of things deliberately apart:

1. **Source**: small text you author and commit (code, configs, docs, per-model
   metadata). The whole system is driven from it.
2. **Assets**: large binaries you download but never commit (model weights, datasets,
   upstream code clones). They are reproducible from the source, so they live only on
   your machine.
3. **Evidence**: what a benchmark produces. This splits in two: a compact run folder
   (manifests + metrics JSON) that is committed, and raw artifacts (per-image
   predictions and logs) that are not.

The git policy follows from this: commit source and compact evidence; keep assets and
raw artifacts local.

- **prepare** вҖ” create each model's Conda env and download its weights + the dataset.
  Writes to local-only `models/*/weights/` and `data/raw/`.
- **smoke** вҖ” push one image through each model to prove the env, config, and
  checkpoint are wired up. Fast. Not a benchmark.
- **run** вҖ” the real thing: evaluate a model on a dataset, producing a **run folder**.
- **aggregate** вҖ” scan every committed run folder's metrics and flatten them into one
  CSV. **report** вҖ” render that CSV as a browsable HTML table.
- aggregate/report are cheap and you can run them locally to preview, but on `main`
  **CI runs them for everyone** so nobody hand-merges a derived file
  (see [collaboration.md](collaboration.md)).

## Runs

A run evaluates one or more models on one or more datasets and writes a folder named by
a unique, timestamped run id, for example:

```
benchmarks/runs/models-yolo26x_pose__bench-coco17_val2017__scope-full5000__20260609T215042Z/
```

The folder is a self-contained record of what was measured, on what, and on which
machine: the command, the models and datasets, the hardware and software, dataset
hashes, the metric numbers, and a per-run HTML summary. Runs are treated as immutable;
to change a result, produce a new run with a new id. This matters because:

- A laptop functional check and an A100 speed run are not comparable unless each number
  carries its own hardware/software context, which the run folder provides.
- Unique names mean two people committing two runs never conflict.
- Not editing a past run keeps committed numbers trustworthy and reviewable.

The full layout of a run folder is in [results-format.md](results-format.md).

## Tracked vs. local

| Path | Committed | What it is |
| ---- | --------- | ---------- |
| `pose_estimation/`, `scripts/`, `tests/` | yes | Library and CLI code |
| `configs/*.yaml` | yes | Models, datasets, metrics, mappings (source of truth) |
| `docs/` | yes | This documentation |
| `models/<id>/model.yaml`, `README.md` | yes | Per-model metadata (generated by `sync_model_store.py`) |
| `benchmarks/runs/<run_id>/` | yes | Compact, immutable benchmark evidence |
| `data/splits/` | yes | Small, stable split definitions (IDs only, no media) |
| `.gitkeep` placeholders | yes | Keep expected-but-empty local dirs in the tree |
| `models/<id>/weights/` | local | Checkpoints (downloaded, multi-GB) |
| `models/<id>/checksums/` | local | Hashes generated from local weights |
| `data/raw/`, `data/derived/` | local | Datasets and intermediate data |
| `external/` | local | Cloned upstream repos (MMPose, OpenPose, ...) |
| `benchmarks/artifacts/<run_id>/` | local | Raw predictions and per-image logs |
| `results/*.csv`, `benchmarks/reports/*.html` | derived | Generated by `aggregate`/`report`; CI publishes them |
| `Papers/` | local | Reference dump; curate notes into `docs/` instead |

`python3 scripts/audit_repo.py --fail` flags any local or generated artifact that was
tracked by mistake. Run it before committing.

## Why each model gets its own Conda environment

The candidate models come from incompatible ecosystems вҖ” MMPose (and a pinned
`mmcv`), Ultralytics, MediaPipe, a CMU OpenPose C++/Caffe build, Sapiens2. They can't
share one Python environment. So `configs/model_envs.yaml` defines an install
**profile** per model, and `setup_model_envs.py` builds one Conda env per model (e.g.
`cricket-rtmw-l`, `cricket-yolo26x-pose`). The benchmark runner shells into the right
env with `conda run -n <env>` automatically вҖ” you don't manage that by hand.

## Where everything lives

```
configs/         Source of truth: model registry, envs, datasets, metrics, mappings
pose_estimation/ Library: adapters, datasets, metrics, schemas, hardware, triangulation, UE export
scripts/         CLI entry points (benchmark.py is the master)
tests/           Unit/integration tests
docs/            This documentation
models/<id>/     Per-model metadata (tracked) + weights/checksums (local)
data/            raw/ + derived/ (local) + splits/ (tracked)
benchmarks/      runs/ (committed evidence) + artifacts/ (local raw) + reports/ (derived)
results/         Derived comparison CSVs (CI-generated)
external/        Upstream source clones (local)
```
